Legacy format
ESA SAFE / ZIP
  • File-based archive (ZIP + XML)
  • Full download required
  • Mission-specific structure
  • Not cloud-optimised
→
New format
EOPF Zarr
  • Chunked, cloud-native arrays
  • Lazy access — no full download
  • Harmonised across missions
  • Works directly with xarray, Python

Access the data programmatically

Recommendation

Work in the cloud via JupyterHub whenever possible. Zarr's chunked format is designed for lazy remote access — direct download is supported but discouraged.

import xarray as xr # Open EOPF Zarr product directly from object store ds = xr.open_dataset( "s3://e05ab01a9d56408d82ac32d69a5aae2a:202506-s02msil2a/...", engine="eopf-zarr", chunks="auto" ) ds["B04"].isel(time=0).plot()
from pystac_client import Client # Search the STAC catalogue catalog = Client.open("https://stac.browser.user.eopf.eodc.eu") items = catalog.search( collections=["sentinel-2-l2a"], bbox=[12.0, 41.5, 13.0, 42.5] ).item_collection()
s5cmd \ --credentials-file ~/.aws/credentials \ cp \ s3://e05ab01a9d56408d82ac32d69a5aae2a:202506-s02msil2a/*.zarr \ ./local_output/

Format & technology

SAFE (Standard Archive Format for Europe) is the current Sentinel product format — a ZIP archive containing XML metadata and binary data files. EOPF Zarr replaces this with a cloud-native, chunked array format based on Zarr v2/v3. The key difference is that Zarr allows lazy access: you can load only the spatial extent or time slice you need, without downloading the entire product.
Zarr is often described as the cloud-native successor to NetCDF. It supports chunked storage natively, works directly with object stores like S3, and integrates cleanly with the Python scientific stack (xarray, Dask, NumPy). Cloud Optimised GeoTIFF (COG) is strong for 2D raster data but handles multi-dimensional EO data less naturally than Zarr.
The Samples Service currently provides products for Sentinel-1 (IW SLC, IW GRD), Sentinel-2 (L1C, L2A), and Sentinel-3 (OLCI L1, SLSTR L1/L2). Coverage is expanding — check the STAC Browser for the current inventory. Sentinel-5P and Sentinel-6 are planned for future releases.
The format specification is stable but still under active development. The Samples Service is an early-access proving ground — its purpose is to validate the format with real users before the full operational rollout. Product Data Format Specifications (PDFS) for S1, S2, and S3 are published and linked from the Data & Tools page. Minor schema changes may occur; the community forum is the best place to track updates.

Data access

Yes. Authentication uses your Copernicus Data Space Ecosystem (CDSE) identity. If you don't have one, registration is free at dataspace.copernicus.eu. The same account grants access to the STAC Browser, JupyterHub, and programmatic API access.
Cloud-native access via xarray and the eopf-zarr backend is strongly recommended. The Analytics Hub (JupyterHub + Dask) provides a pre-configured environment with all dependencies installed. Direct download via s5cmd or ZIP is available but should be a last resort.
Use the STAC Browser at stac.browser.user.eopf.eodc.eu to explore the catalogue interactively. For programmatic access, the STAC API is queryable via pystac-client — see the Data & Tools page for a copy-ready example. Products are organised by mission, date, and geographic area.

About

What is EOPF Zarr?

The Earth Observation Processing Framework (EOPF) is ESA's initiative to modernise how Copernicus Sentinel data is stored, distributed, and processed.

The technology

Why Zarr?

What Zarr is

Zarr is a community-maintained, open format for storing large, chunked N-dimensional arrays. It is the cloud-native successor to NetCDF and an alternative to COG that handles multi-dimensional data more naturally.

Why ESA chose it

Zarr supports lazy I/O — users access only the spatial or temporal slice they need, with no full download. This makes it ideal for object store deployment and directly compatible with xarray, Dask, and NumPy.

What EOPF adds

EOPF defines a common metamodel layered on top of Zarr, harmonising product structure across all Sentinel missions. A single code pattern opens S1, S2, or S3 products without mission-specific workarounds.

Where this is going

This service is an early-access proving ground. The full transition of operational Copernicus products is funded and in progress. The service exists to validate the format with real users before the full rollout.

Architecture

How data is reformatted

Existing CDSE products are converted automatically via the EOPF CPM library and published to a STAC catalogue.

Query CDSE

Download SAFE products over a configurable area of interest

Step 2
Convert

EOPF CPM library reformats SAFE → Zarr

Step 3
Upload

Zarr stores written to EODC S3 object storage

Step 4
Catalogue

STAC items generated and registered in the API

Security

The STAC API and STAC Browser run behind an API Gateway. Authentication is handled via Copernicus Data Space Ecosystem (CDSE) identities.
